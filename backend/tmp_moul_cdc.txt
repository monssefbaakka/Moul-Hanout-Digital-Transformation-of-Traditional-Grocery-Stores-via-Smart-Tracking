Cahier des charges — Application web Moul l Hanout 
Synthèse exécutive 
Moul l Hanout est une application web de gestion opérationnelle d’un commerce de proximité 
(hanout) visant à : (1) fluidifier la vente (POS web), (2) fiabiliser le stock (mouvements, alertes), 
(3) simplifier la gestion du catalogue et des prix, et (4) donner une visibilité de pilotage 
(rapports essentiels). Le présent cahier des charges est rédigé dans un style académique et 
professionnel, avec une structuration “clean work” (périmètre P0/P1/P2, exigences 
non‑fonctionnelles mesurables, architecture cible, maquettes, modèles de données, tests, 
planning, budget, risques, SLA, gouvernance). 
Connecteurs activés (listés) : Figma. 
- Le connecteur permet ici de générer des diagrammes FigJam (Mermaid supporté : flowchart, 
sequence, state, gantt). 
- Il ne permet pas (dans cette configuration) de “lister des fichiers” ni de “créer des frames 
design” sans fournir un fileKey/nodeId d’un fichier Design existant. En conséquence : les 
maquettes web sont fournies en images (à importer dans Figma comme frames), et les 
diagrammes sont fournis via liens FigJam. 
Hypothèses non spécifiées (explicitement signalées) : 
- public cible exact ; 
- budget maximal ; 
- contraintes légales locales (selon pays de déploiement). 
Ces points doivent être confirmés en cadrage ; le CDC ci‑dessous est néanmoins opérationnel et 
directement exploitable pour lancer un MVP web. 
Informations nécessaires et cadrage documentaire 
Besoins d’information pour répondre correctement 
Pour un CDC solide et “exécutable”, les informations suivantes sont déterminantes : 
• Profil(s) utilisateurs (gérant, caissier, multi‑boutiques), niveau de maturité numérique, 
langues (FR/AR), et matériel (PC, tablette). 
• Flux métier prioritaires (vente, produits, stock, réassort, rapports) et fréquence d’usage. 
• Contraintes légales et de conformité (pays d’exploitation, conservation des données, 
hébergement, fiscalité). 
• Objectifs de service (SLA, support, horaires, criticité opérationnelle en caisse). 
• Budget maximal et modèle de réalisation (équipe interne / prestataire / mix). 
• Contraintes techniques (hébergeur imposé, SI existant, intégrations attendues).

Objectifs du produit et périmètre fonctionnel 
Objectifs du produit 
Objectifs métier (KPI suggérés) : - Accélération de la vente : réduire le temps moyen de 
transaction (TMT) et le nombre d’erreurs (retours/annulations). 
- Fiabilisation du stock : limiter écarts stock théorique vs réel, améliorer la réactivité sur 
ruptures. 
- Pilotage simplifié : faire émerger CA, transactions, top produits, marges (si coût renseigné) 
sans compétences avancées. 
Objectifs qualité : - Performance web mesurée (Core Web Vitals). 
- Sécurité applicative de niveau professionnel (référentiels OWASP). 
- Accessibilité conforme (WCAG ; RGAA si contexte France). 
Périmètre fonctionnel P0/P1/P2 
Définitions : 
- P0 (MVP) : indispensable pour délivrer la valeur et lancer en production. 
- P1 (Post‑MVP) : améliore fortement la solution, sans bloquer le lancement. 
- P2 (Évolutions) : extensions selon budget/retours terrain.

Cas d’utilisation et règles métier (extraits) 
• Vente : une vente validée doit créer une transaction + lignes, décrémenter le stock, et 
consigner l’action (audit). 
• Stock : tout changement de stock doit passer par un mouvement (traçabilité). 
• Permissions : seules certaines actions (prix, stock, exports, gestion utilisateurs) sont 
autorisées pour l’Owner.

Exigences non‑fonctionnelles mesurables 
Performance (Core Web Vitals) 
Cibles recommandées (niveau “bon”) : - LCP ≤ 2,5 s, INP ≤ 200 ms, CLS ≤ 0,1 (référence Core 
Web Vitals, web.dev). 
- Mesure au 75e percentile, séparée mobile/desktop. 
Exigences complémentaires : - Budget performance (ex. JS initial ≤ X Ko gzip, à définir) et 
stratégies d’optimisation (lazy loading, code-splitting). 
- CDN pour assets, compression, cache HTTP, images optimisées. 
Sécurité (OWASP ASVS / MASVS) 
Référentiels : - OWASP ASVS pour le web/backend/API (contrôles d’authentification, gestion de 
session, contrôle d’accès, validation des entrées, logs, etc.). 
- OWASP MASVS à considérer si PWA/usage intensif sur navigateur mobile (stockage local, 
transport, intégrité). 
Exigences sécurité (testables) : - Contrôles d’accès exclusivement côté serveur (aucune 
permission “client-side only”). 
- Journal d’audit immuable au niveau applicatif (écriture append‑only logique). 
- Protection brute force, politique mots de passe, gestion des sessions (rotation, expiration). 
- Validation systématique des entrées, protection XSS/CSRF, CSP dès P0. 
- Gestion des secrets (vault/variables sécurisées), rotation et principe du moindre privilège. 
Accessibilité (WCAG / RGAA) 
Références : - W3C WCAG 2.2 cible AA (recommandation pour applications pro). 
- RGAA (France) : méthode et critères de vérification (à appliquer si contexte réglementaire FR). 
Exigences : - Navigation clavier complète (tab/focus), labels explicites, messages d’erreur 
accessibles. 
- Contraste et taille de police adaptés ; composants réutilisables “accessibles by default”. 
Compatibilité navigateurs/OS 
Cibles (hypothèse raisonnable) : - Chrome/Edge/Firefox : dernières 2 versions majeures. 
- Safari macOS : dernière version majeure. 
- Tablettes : iPadOS Safari récent, Android Chrome récent. 
Scalabilité et résilience 
• Architecture stateless pour l’API, scalabilité horizontale. 
• Contention DB limitée par indexation, pagination, agrégations pré‑calculées pour rapports. 
• Sauvegardes automatisées, tests de restauration, monitoring et alerting.

Architecture technique proposée et schémas de données 
Architecture cible (logique MVP → évolutif) 
Proposition “web moderne, maintenance simple” : - Front : SPA/SSR (TypeScript) + design system 
minimal. 
- Backend : API REST documentée OpenAPI, RBAC, validations. 
- DB : PostgreSQL (transactions et cohérence). 
- Cache : Redis (sessions/rate‑limit/caches). 
- Stockage : objet (photos, exports). 
- Observabilité : logs structurés + métriques. 
API : - Spécification par **OpenAPI Initiative OpenAPI 3.1.1 (contrat, tests, documentation). 
- Versioning /v1 ; conventions d’erreurs ; idempotence (ex. mouvements stock). 
Hébergement (options, pas de contrainte imposée) : - PaaS/containers managés (time‑to‑market) 
; 
- Kubernetes managé (si multi‑tenant large) ; 
- VPS + containers (budget minimal mais exploitation plus lourde). 
IaC : - Référence de bonnes pratiques SDLC sécurisé : **NIST SSDF (circuits de build, 
dépendances, contrôles). 
- Terraform/Pulumi/équivalent ; séparation dev/staging/prod ; secrets gérés proprement.

Critères d’acceptation et stratégie de tests 
Stratégie de tests (vérifiable) : - Unitaires : calculs (totaux, rendu monnaie, seuils stock, remises). 
- Intégration : API + DB (vente → lignes → mouvements stock). 
- E2E : scénarios “vendre”, “réassort”, “gérer produit”, “droits”. 
- Sécurité : check‑list ASVS (auth, session, authz, validation, logs) + MASVS si PWA. 
- Accessibilité : audits WCAG + méthode RGAA si applicable. 
- Performance : budgets + mesures LCP/INP/CLS.
