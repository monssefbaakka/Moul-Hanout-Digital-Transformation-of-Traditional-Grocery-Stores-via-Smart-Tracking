# 🏪 Moul Hanout — Digital Transformation of Traditional Grocery Stores

> Production-grade POS SaaS platform for traditional Moroccan grocery stores.
> Built with NestJS, Next.js 15, PostgreSQL, and Prisma.

## 🚀 Quick Start (Development)

### Prerequisites
- Node.js 20+
- Docker Desktop
- Git

### 1. Clone & Setup

```bash
git clone <repo-url>
cd Moul-Hanout-Digital-Transformation-of-Traditional-Grocery-Stores-via-Smart-Tracking

# Copy all .env files
cp .env.example .env
cp backend/.env.example backend/.env
cp frontend/.env.example frontend/.env.local

# Edit backend/.env — set DATABASE_URL, JWT_SECRET, JWT_REFRESH_SECRET
```

### 2. Start the Database

```bash
# Start Postgres + Redis via Docker
docker-compose up postgres redis -d
```

### 3. Start the Backend

```bash
cd backend
npm install

# Run Prisma migrations
npx prisma migrate dev --name init

# Seed the database
npx ts-node prisma/seeds/seed.ts

# Start with hot reload
npm run start:dev
```

> 🔗 API: http://localhost:4000/api/v1
> 📚 Swagger: http://localhost:4000/api/docs

### 4. Start the Frontend

```bash
cd frontend
npm install
npm run dev
```

> 🔗 App: http://localhost:3000

## 🗃️ Database Migrations

```bash
# Create a new migration (name it after what changed)
./scripts/migrate.sh dev "add_product_tags"

# OR manually:
cd backend && npx prisma migrate dev --name add_product_tags

# Check migration status
./scripts/migrate.sh status

# Open Prisma Studio (GUI)
./scripts/migrate.sh studio
```

### ⚠️ Team Migration Rules

1. **Never edit** a migration file after it's been committed
2. **One migration owner**: When two devs need schema changes simultaneously,
   communicate via a `#db-changes` channel, merge one at a time
3. **Always** run `npx prisma generate` after pulling schema changes
4. **Review** `schema.prisma` carefully in PRs — it's the single source of truth

## 🌿 Branching Strategy

```
main            ← production-ready, protected branch
├── develop     ← integration branch (both devs merge here)
│   ├── feat/backend/auth-refresh-token    ← backend feature
│   ├── feat/frontend/product-scanner      ← frontend feature
│   ├── fix/backend/stock-expiry-job       ← backend fix
│   └── fix/frontend/cart-total-rounding   ← frontend fix
```

### Rules
- Branch naming: `feat|fix|chore/[backend|frontend|shared]/short-description`
- PRs always target `develop`, not `main`
- Deploy to production: `develop` → `main` (squash + tag)
- Minimum 1 reviewer per PR
